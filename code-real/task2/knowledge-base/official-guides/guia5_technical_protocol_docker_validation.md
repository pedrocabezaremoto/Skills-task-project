# Technical Protocol for Real Coder Docker Validation

## 1. Strategic Context and Executive Overview
Maintaining a containerized validation environment is a strategic imperative for the rigorous evaluation of software submissions. The "Real Coder" workflow is designed to standardize the evaluation process, ensuring that all code submissions are validated against hidden test suites within an immutable environment. This approach is critical to mitigate "it works on my machine" discrepancies, providing a consistent baseline where the environment—not the host system's configuration—dictates the outcome.

Strict adherence to the Dockerfile and script protocols is mandatory to maintain the integrity of the evaluation pipeline. We must orchestrate dependencies at build-time and enforce directory structures to ensure the telemetry generated by the validation scripts remains accurate. This protocol serves as the operational contract between the code submission and the automated scoring system, ensuring the reliability of the entire validation chain.

The following sections define the infrastructure prerequisites and governance rules for local environment readiness.

## 2. System Infrastructure Prerequisites
Local host readiness is the foundation of the validation pipeline; any configuration drift at the system level will invalidate downstream telemetry. This protocol supports both macOS and Windows environments via Docker Desktop.

### Platform-Specific Infrastructure Requirements

| Feature | macOS (Intel & Apple Silicon) | Windows (AMD64 & ARM64) |
|---|---|---|
| Software | Docker Desktop for Mac | Docker Desktop for Windows |
| Installer Type | Apple Silicon (M1-M4) vs. Intel Chip | AMD64 vs. ARM64 Installer |
| Backend | Native Hypervisor | WSL2 Backend (Mandatory) |
| Terminal | Applications > Utilities > Terminal | PowerShell, Git Bash, or WSL Ubuntu |

### Verification Protocol
The agent must execute the following commands to confirm the Docker daemon constitutes a stable root for the validation chain:
* **Version Check**: `docker --version` (Expected: Docker version 27.2.0 or higher).
* **Daemon Integrity**: `docker info --format '{{.ServerVersion}}'`.
* **Active Status**: If the terminal returns "command not found," launch the Docker Desktop GUI and ensure the "whale" icon is active in the system tray.

The host environment must be verified as stable before attempting any Dockerfile modifications or script executions.

## 3. Dockerfile Governance: Allowed vs. Forbidden Configurations
The Dockerfile functions as a technical contract. Unauthorized modifications to the image definition will break the orchestration layer and invalidate test results.

### Modification Governance

**Allowed Modifications**
* [x] **System Packages**: Adding packages in the "SYSTEM DEPENDENCIES" section via `apt-get install`.
* [x] **Python Dependencies**: Installing libraries using the specific venv path: `/opt/venv/bin/pip install`.
* [x] **Hardcoded Requirements**: Specifying versions for `pyyaml`, `requests`, or `pytest` to align with the codebase.

**Forbidden Modifications**
* [ ] **Base Image**: Never change the `FROM` instruction; use the specific Ubuntu version (22.04 or 24.04) provided in your task taxonomy.
* [ ] **Code Injection**: Never use `COPY` or `ADD` for project code; all codebase assets are volume-mounted at runtime.
* [ ] **Structure**: Do not alter `WORKDIR /app`, `ENV` variables, or existing `RUN` order.
* [ ] **Entrypoints**: Changing the default `CMD ["/bin/bash"]` is strictly prohibited.

**Safe Addition Example (Python 3.14 Environment)**
To maintain the integrity of the specialized venv structure, use the following syntax for adding dependencies:
```dockerfile
# SYSTEM DEPENDENCIES: Add build-essential for C-extensions
RUN apt-get update && apt-get install -y \
    build-essential \
    && rm -rf /var/lib/apt/lists/*

# VENV DEPENDENCIES: Use specific pathing for Python 3.14
RUN /opt/venv/bin/pip install --no-cache-dir \
    numpy \
    pandas \
    scipy
```

## 4. Directory Architecture and File Orchestration
The `validation.sh` script utilizes a rigid folder hierarchy to mount volumes and parse telemetry correctly. Deviation from this architecture results in fatal runtime errors.

### Directory Tree Model
The local environment must be structured as follows:
```text
your-real-coder-project/
├── validation.sh         # Parent: The evaluation orchestration script
└── app/                  # Required subfolder for volume mounting
    ├── Dockerfile        # Environment specification
    ├── tests.zip         # Hidden test suite
    ├── codebase.zip      # The agent's Golden Patch submission
    ├── run.sh            # Protected test runner
    └── parsing.py        # Protected result parser
```

### The "Zip Nesting Guard" Constraint
The orchestration logic includes a hardcoded variable: `MAX_ZIP_DEPTH=3`.
* **The "So What?"**: If the files within `tests.zip` or `codebase.zip` are nested deeper than three levels (e.g., `root/folder/subfolder/file.py`), the extraction logic will fail to locate the entry points. This causes a total failure of the injection layer, resulting in an empty `/app` directory during testing.

## 5. Validation Execution Protocol: Baseline and Verification
The validation process follows a "Before vs. After" methodology to generate the required telemetry for submission.

### Execution Steps
1. **Permission Hardening**: Execute `chmod +x validation.sh`.
2. **APP_DIR Configuration**: Update the `APP_DIR` variable in `validation.sh`.
   * *Pro-tip*: Use `APP_DIR="$(pwd)/app"` if executing from the project root.
3. **Windows-Specific Logic**: If the script fails on Windows, you must manually add the following injection logic to the execution block:
   ```bash
   /bin/bash -c 'parse_results stdout.txt stderr.txt before.json' || true
   /bin/bash -c 'parse_results stdout.txt stderr.txt after.json' || true
   ```
4. **Launch**: Execute `bash validation.sh`.

### The Output Contract
A successful validation run must produce exactly six files in the `app/` directory:
* `before_stdout.txt` & `before_stderr.txt`: Log telemetry from the baseline.
* `before.json`: The Baseline Contract (Tests must **FAIL** on an empty codebase).
* `after_stdout.txt` & `after_stderr.txt`: Log telemetry from the verification.
* `after.json`: The Verification Contract (Tests must **PASS** after the Golden Patch is applied).

*Updating the Dockerfile triggers a `docker build -t real-coder-env .` command, refreshing the local image cache to incorporate new dependencies.*

## 6. System Troubleshooting and Error Mitigation
Proactive mitigation of runtime failures is essential for maintaining the automated validation chain.

### Diagnostic Guide
* **Docker Daemon Failures**: Ensure the "whale" icon is steady. On Windows, verify WSL2 is the default backend. Restarting Docker Desktop often resolves "daemon not found" errors.
* **Permission Denied**: On macOS, grant Full Disk Access to Docker if prompted. Re-apply `chmod +x` to all `.sh` files if they fail to execute.
* **Slow Silicon Mac Builds**: The initial build on Apple Silicon is computationally intensive; subsequent runs will utilize the image cache.
* **Extraction Failures**: If the `after.json` shows no tests run, verify that your `codebase.zip` does not exceed the `MAX_ZIP_DEPTH=3` limit.
* **Windows Script Stalls**: If the script finishes without generating JSON, ensure the `parse_results` bash commands mentioned in Section 5 were correctly injected.

Agents must follow these protocols exactly to ensure the submission is compatible with the automated evaluation pipeline. Non-compliance results in an automatic invalidation of the task.
